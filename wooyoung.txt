# Git & GitHub 상세 학습 정리

## 1. 초기 설정 (Setup)
Git을 설치하고 가장 먼저 해야 할 일입니다. 커밋을 할 때 누가 했는지 기록하기 위해 필요합니다.

```bash
git config --global user.name "geonha"
git config --global user.email "geonha0115@naver.com"
```

> 💡 **`--global` 옵션이란?**
> `--global`을 붙이면 현재 컴퓨터에서 수행하는 모든 Git 프로젝트에 이 사용자 정보가 공통으로 적용됩니다. 만약 특정 프로젝트에서만 다른 이름/이메일을 쓰고 싶다면, 해당 폴더 안에서 `--global`을 빼고 명령어를 입력하면 됩니다.

**설정 확인 및 도움말**
*   `git config --list` : 현재 설정된 모든 내용을 확인할 수 있습니다.
*   `git help [명령어]` : 해당 커맨드의 자세한 사용법(공식 매뉴얼)이 나옵니다. (종료는 `q`) macOS나 리눅스 환경이라면 `man git-[명령어]`를 사용해도 동일한 매뉴얼을 볼 수 있습니다.

## 2. Git의 3가지 작업 영역 (Core Concepts)
Git은 내부적으로 크게 3가지 종류의 작업 영역을 두고 동작합니다. 이 개념을 확실히 이해해야 `add` 의 필요성을 알 수 있습니다.

```mermaid
graph LR
WD["Working Directory<br>(작업 공간/Working Tree)"] -- "git add" -->
SA["Staging Area<br>(임시 저장/Index)"]
SA -- "git commit" --> REPO["Repository<br>(저장소/HEAD)"]

style WD fill:#f9f,stroke:#333,stroke-width:2px
style SA fill:#bbf,stroke:#333,stroke-width:2px
style REPO fill:#bfb,stroke:#333,stroke-width:2px
```

1.  **Working Directory (워킹 디렉토리)**: 'Working Tree'라고도 부릅니다. 작업을 하는 프로젝트 디렉토리 그 자체를 말합니다. (예: MathTool 디렉토리)
2.  **Staging Area (스테이징 에어리어)**: 'Index'라고도 부릅니다. `git add` 를 한 파일들이 존재하는 영역입니다. 커밋을 하게 되면 이 영역에 있는 파일들만 커밋에 반영됩니다.
3.  **Repository (레포지토리)**: Working Directory의 변경 이력들이 저장되어 있는 영역입니다. 커밋들이 저장되는 영역이며, 마치 영화의 한 장면(스냅샷, Snapshot)처럼 저장됩니다. 실제로는 프로젝트 폴더 내의 숨겨진 `.git` 디렉토리가 바로 이 레포지토리입니다.

> 💡 **왜 Staging Area가 필요한가? (상세 예시)**
> Working Directory에서 바로 커밋하지 않고 왜 중간 단계를 거칠까요?
> 예를 들어 `A.txt`와 `B.txt`를 모두 수정했습니다.
> *   `A.txt` : 기능 구현 완료
> *   `B.txt` : 버그 수정 중 (아직 미완성)
>
> 이때 `A.txt`의 최신 모습만 커밋에 포함시키고, `B.txt`는 포함시키고 싶지 않을 수 있습니다. 만약 Staging Area가 없다면 수정한 모든 파일이 커밋되어야 하므로 이런 선별적인 버전 관리가 불가능해집니다. 원하는 것만 골라서(`git add`) 커밋하기 위해 존재하는 영역입니다.

## 3. 파일의 상태 (File Lifecycle)
Git으로 관리되는 파일은 일종의 '상태(Status)'를 가집니다. 크게 Untracked와 Tracked로 나뉩니다.

```mermaid
stateDiagram-v2
[*] --> Untracked
Untracked --> Staged: git add (최초 추적)
Staged --> Unmodified: git commit
Unmodified --> Modified: 파일 내용 수정
Modified --> Staged: git add
Modified --> Unmodified: git restore (변경 취소)
```

1.  **Untracked (추적되지 않음)**: 파일을 새로 생성하고 한 번도 `git add` 해주지 않은 상태. Git이 이 파일의 변동사항을 전혀 신경 쓰지 않습니다.
2.  **Tracked (추적됨)**: Git이 변동사항을 추적하고 있는 상태입니다.
    *   **Staged**: 파일 내용이 수정되고 나서, Staging Area에 올라와 있는 상태. (커밋 준비 완료)
    *   **Unmodified**: 최신 커밋과 비교했을 때 바뀐 게 없는 상태. 커밋 직후에는 모든 파일이 이 상태가 됩니다.
    *   **Modified**: 최신 커밋과 비교했을 때 조금이라도 바뀐 내용이 있는 상태.

### 🚫 추적 예외 처리 (.gitignore)
보안상 민감한 파일(API 키 등)이나 불필요한 자동 생성 파일(로그 모듈, 라이브러리 폴더 등)은 Git이 추적하지 않게 만들어야 합니다.
프로젝트 최상단에 `.gitignore` 파일을 만들고 무시할 파일명이나 폴더 규칙을 적어두면, Git이 해당 파일들을 'Untracked' 상태로도 보지 않고 업로드 대상에서 완전히 제외합니다.

## 4. 핵심 명령어 모음 (Commands)

### 시작 및 조회 (Start & View)
| 명령어 | 설명 |
| :--- | :--- |
| `git init` | 현재 디렉토리를 Git이 관리하는 프로젝트로 설정 (`.git` 생성) |
| `git clone [주소]` | 기존 원격 저장소(GitHub 등)를 내 로컬 컴퓨터로 복제해 오기 |
| `git status` | 현재 파일들의 상태(Untracked, Modified, Staged 등) 확인 |
| `git log` | 커밋 내역 확인 (최신이 위쪽) |
| `git log --pretty=oneline` | 커밋 히스토리를 한 줄로 깔끔하게 확인 |
| `git show [커밋ID]` | 해당 커밋의 상세 변경 내용을 확인 (ID는 앞 4자리만 써도 됨) |
| `git diff [커밋A] [커밋B]` | 두 커밋 간의 차이 비교 |

### 작업 흐름 (Workflow)
| 명령어 | 설명 |
| :--- | :--- |
| `git add [파일]` | 특정 파일을 Staging Area로 올리기 |
| `git add [디렉토리]` | 해당 디렉토리 내 수정사항 있는 모든 파일을 올리기 |
| `git add .` | Working Directory 내의 모든 수정 파일을 Staging Area로 올리기 |
| `git commit -m "메시지"` | Staging Area의 내용을 커밋(버전)으로 남기기 |
| `git commit` | `-m` 옵션 없이 실행 시 텍스트 에디터가 열려 긴 메시지 작성 가능 |
| `git commit --amend` | 최신 커밋 수정. 메시지를 고치거나 누락된 파일을 추가하여 덮어쓰기 |

### 브랜치 및 병합 (Branch & Merge)
| 명령어 | 설명 |
| :--- | :--- |
| `git branch` | 현재 존재하는 브랜치 조회 |
| `git branch [이름]` | 새로운 브랜치 생성 |
| `git branch -d [이름]` | 브랜치 삭제 |
| `git switch [이름]` | 해당 브랜치로 이동 (최신 권장, 기존 `checkout` 대체) |
| `git switch -c [이름]` | 브랜치 생성과 동시에 이동 (기존 `checkout -b` 대체) |
| `git merge [이름]` | 현재 브랜치에 다른 브랜치를 병합 |
| `git merge --abort` | 병합 중단 (충돌 발생 시, 머지 시도 전으로 되돌림) |

### 되돌리기 (Undo)
| 명령어 | 설명 |
| :--- | :--- |
| `git restore [파일]` | Working Directory의 파일 수정 내역 취소 (최신 커밋 상태로 복구) |
| `git restore --staged [파일]` | Staging Area에 올린 파일을 내림 (명령어 분리, 기존 `reset [파일]` 대체) |
| `git reset [옵션] [커밋]` | 특정 커밋 시점으로 완전히 되돌리기 (과거로 이동) |
| `git revert [커밋]` | 특정 커밋의 변경을 취소하는 새로운 커밋 생성 (안전한 되돌리기) |

### 원격 저장소 제어 (Remote)
| 명령어 | 설명 |
| :--- | :--- |
| `git remote add origin [주소]`| 로컬 저장소에 원격 저장소(`origin`) 주소 지정 및 연결 |
| `git push [원격] [로컬브랜치]`| 로컬 커밋 내용을 원격 저장소에 업로드 (예: `git push origin master`) |
| `git pull` | 원격 저장소의 최신 변경 내용을 로컬로 받아와서 자동 병합 |
| `git fetch` | 원격 저장소의 변경 내역만 가져와 상태 업데이트 (병합은 안 함) |

## 5. 좋은 커밋 남기기 (Commit Guidelines)
커밋 메시지는 협업의 핵심 소통 수단입니다. 일반적인 가이드라인은 다음과 같습니다.

### ✍ 커밋 메시지 작성 가이드라인
1.  **제목과 본문 분리**: 커밋 메시지의 제목과 상세 설명 사이에는 한 줄을 비워두세요. Git의 일부 명령어는 이 공백을 기준으로 제목과 본문을 구분합니다.
2.  **제목 규칙**:
    *   제목 뒤에 온점(.)을 붙이지 마세요.
    *   제목의 첫 번째 알파벳은 대문자로 작성하세요.
    *   제목은 명령조(Imperative mood)로 작성하세요. (예: `Fix it`, `Update code` - "고쳤다"가 아니라 "고쳐라" 느낌)
3.  **상세 내용(Body) 작성 팁**:
    *   '무엇'을 했는지보다 '왜' 커밋을 했는지, '어떤 문제'가 있었는지, '해결책의 효과' 등을 적습니다.
    *   다른 사람들이 내 코드를 모른다고 가정하고 최대한 친절하게 작성하세요.

> 💡 **Vim 에디터 사용법 (Commit 메시지 작성 창)**
> `git commit` 뒷부분에 `-m` 옵션을 생략하면 긴 메시지를 적기 위해 Vim 에디터가 열립니다.
> 1. `i` 키를 눌러 하단에 상태가 바귀면 **입력 모드(Insert)** 진입.
> 2. 제목 작성 -> 엔터 두 번 -> 본문 작성.
> 3. `ESC` 키를 눌러 입력 모드 종료.
> 4. `:wq` 입력 후 엔터 (Write & Quit, 저장 후 종료).

### 🛠 커밋 수행 가이드라인 (Best Practices)
1.  **Atomic Commit (원자적 커밋)**: 하나의 커밋에는 하나의 수정사항(하나의 이슈)만 담도록 하세요.
    *   예를 들어 함수 A 삭제, 함수 B 수정, 함수 C 삭제를 한 번에 커밋하면, 나중에 에러가 났을 때 원인 파악이 힘듭니다.
    *   최대한 작은 단위의 변화를 기준으로 커밋해야 나중에 이력을 파악하기 쉽습니다.
2.  **Runnability (실행 가능성)**: 커밋된 상태의 프로젝트는 전체 코드를 실행했을 때 에러가 발생하지 않는 상태여야 합니다.
    *   과거의 커밋으로 돌아가서 작업을 하거나, 그 지점에서 새로운 프로젝트를 시작할 수도 있기 때문입니다.

## 6. Reset과 Checkout의 원리 (Deep Dive)
단순히 명령어를 외우는 것을 넘어, `HEAD`와 `Branch`의 관계를 이해해야 합니다.

*   **HEAD**: 현재 작업 중인 브랜치를 가리키는 포인터입니다.
*   **상대 참조**: 커밋 ID를 매번 찾기 귀찮을 때 사용합니다.
    *   `HEAD^` : 바로 이전 커밋 (1단계 전)
    *   `HEAD~2` : 2단계 전 커밋

### 💡 Reset vs Checkout 비교 (중요!)
| 구분 | 원리 | 결과 | Detached HEAD |
| :--- | :--- | :--- | :--- |
| `git reset` | `HEAD`가 가리키는 브랜치가 다른 커밋을 가리키도록 만듦 | 브랜치의 끝(최신 상태)이 바뀜 | 발생하지 않음 |
| `git checkout` (또는 `switch`) | `HEAD` 포인터 자체가 다른 커밋이나 브랜치를 가리키도록 만듦 | 브랜치는 그대로 두고, 내가 보는 시점만 바뀜 | 발생 가능 |

**Detached HEAD란?**
브랜치를 통하지 않고 커밋을 직접 가리키는 상태.
`git checkout [커밋ID]` 를 하면 `HEAD`가 브랜치에서 떨어져 나와 특정 커밋을 직접 가리키게 됩니다. 주로 과거의 특정 시점으로 돌아가서 새로운 실험(새 브랜치 생성)을 하고 싶을 때 사용합니다.

### 💥 Reset의 3가지 옵션 상세
`git reset [옵션] [돌아갈 커밋]` 실행 시 3가지 영역이 어떻게 변하는지 정리했습니다.

1.  `--soft` :
    *   Repository(HEAD): 과거 커밋으로 이동.
    *   Staging Area: 유지됨 (커밋하기 전 상태로 돌아감).
    *   Working Directory: 유지됨.
2.  `--mixed` (기본값):
    *   Repository(HEAD): 과거 커밋으로 이동.
    *   Staging Area: 리셋됨 (add 하기 전 상태로 돌아감).
    *   Working Directory: 유지됨. (작업 내용은 살아있음)
3.  `--hard` :
    *   Repository(HEAD): 과거 커밋으로 이동.
    *   Staging Area: 리셋됨.
    *   Working Directory: 리셋됨. (작업하던 내용까지 모두 지워지고 과거 상태로 완벽히 돌아감)

## 7. 브랜치와 병합 (Branch & Merge)

### 🌿 병합(Merge)의 종류
병합은 `HEAD` 커밋에 다른 브랜치의 커밋을 합쳐 새로운 상황을 만드는 것입니다.

1.  **Fast-forward Merge (빨리감기 머지)**:
    *   두 브랜치가 커밋 히스토리 상에서 같은 선(line) 상에 있을 때 발생합니다.
    *   새로운 커밋이 생기지 않고, `HEAD` 포인터만 최신 커밋으로 이동합니다.
2.  **3-Way Merge (3방향 머지)**:
    *   두 브랜치가 갈라져서 서로 다른 커밋 내역을 가질 때 발생합니다.
    *   (1) 공통 조상(Base), (2) 내 브랜치, (3) 상대 브랜치 3개의 커밋을 비교하여 새로운 머지 커밋을 만들어냅니다.

### 📊 3-Way Merge의 작동 원리 (Git의 선택 기준)
Git은 Base(공통 조상)와 비교하여 변화가 있는 쪽을 채택합니다.

| 경우 | Base (조상) | Master (나) | Premium (상대) | 결과 (채택) | 설명 |
| :---: | :---: | :---: | :---: | :---: | :--- |
| Case 1 | A | A | B | B | Base와 비교해 변화가 있는 쪽을 채택 |
| Case 2 | 1 | 2 | 1 | 2 | Base와 비교해 변화가 있는 쪽을 채택 |
| Case 3 | "Hello" | (공백) | "Hello" | (공백) | 삭제된 것도 변화로 인정하여 채택 |
| Case 4 | "Bye" | "Fighting" | "Please" | 🚨 Conflict | 둘 다 변화가 일어남 -> Git이 판단 포기 |

### 🛡 충돌(Conflict) 해결 전략
Case 4처럼 두 브랜치에서 같은 파일의 같은 부분을 수정했을 때 충돌이 발생합니다.

1.  **단일 파일 충돌**:
    *   파일을 열어 `=======` 기준 위쪽(내 것), 아래쪽(상대 것)을 확인합니다.
    *   원하는 코드로 수정한 뒤 저장합니다.
    *   `git add [파일]` -> `git commit` 으로 해결합니다.
2.  **다중 파일 충돌**:
    *   `git status` 로 충돌난 파일 목록을 확인합니다.
    *   하나씩 수정하고 `git add` 하거나, 한 번에 다 수정하고 모두 해결되었으면 커밋합니다.
3.  `git merge --abort` :
    *   충돌이 너무 많거나 지금 해결하기 힘들 때 사용합니다.
    *   머지 시도 자체를 취소하고 머지 이전 상태로 깔끔하게 돌아갑니다.

## 8. 원격 저장소와 협업 (Remote & Collaboration)

### ☁ Origin과 Upstream의 정확한 의미

1.  **Origin (기원/근원)**:
    *   `git remote add origin [주소]`
    *   리모트 레포지토리의 주소를 가리키는 별명(Alias)입니다.
    *   왜 하필 origin인가? 내 컴퓨터로 코드를 가져온 입장에서 보면, 그 리모트 저장소가 코드의 '근원'이기 때문입니다. 관례적으로 사용하지만 다른 이름을 써도 됩니다.
2.  **Upstream (Tracking Connection)**:
    *   `git push -u origin master` (`-u` 는 `--set-upstream` 의 약자)
    *   로컬 브랜치와 리모트 브랜치를 연결(Tracking)하는 설정입니다.
    *   이 연결이 맺어지면(Tracking Connection), 이후에는 단순히 `git push`, `git pull` 이라고만 입력해도 Git이 알아서 짝을 찾아 보냅니다.
    *   옵션을 안 쓰면? `git push origin master:master` 처럼 매번 목적지를 다 적어줘야 합니다.
3.  **origin/master**:
    *   내 컴퓨터에 있지만, 실제로는 리모트 저장소의 master 브랜치 상태를 보여주는 브랜치입니다.
    *   `git push` 나 `git pull` 을 해야만 내 로컬 `master` 와 위치가 동기화됩니다.

### 💡 Fetch vs Pull (가져오기의 차이)
`git pull` 은 사실 `git fetch` + `git merge` 를 합친 명령어입니다.

| 구분 | `git fetch` | `git pull` |
| :--- | :--- | :--- |
| **동작** | 리모트 브랜치의 최신 커밋을 가져오기만 함 | 가져와서 내 로컬 브랜치와 자동으로 병합(Merge)함 |
| **상태** | 로컬 코드는 변하지 않음 (`origin/master` 만 갱신) | 로컬 코드가 최신 내용으로 바뀜 |
| **용도** | 무작정 합치기 전에, 변경 내용을 먼저 확인 (`git diff origin/master`) 하고 싶을 때 안전하게 가져오기 | 그냥 최신 내용을 빨리 반영하고 싶을 때 |

*   충돌이 걱정될 때는 `git fetch` 를 먼저 하고, `git diff master origin/master` 로 차이를 확인한 뒤 수동으로 머지하는 것이 안전합니다.

### 👥 협업 프로세스 (Collaborator 초대)
내 저장소(Remote)에 다른 사람이 push 할 수 있게 하려면 권한을 줘야 합니다.

1.  GitHub 레포지토리 상단 **Settings** 탭 클릭.
2.  좌측 바에 있는 **Collaborators** 메뉴 클릭 (Access 섹션 하위).
3.  **Add people** 버튼 클릭 -> 사용자 아이디나 이메일을 검색 후 초대.
4.  상태: Pending (초대 대기 중).
5.  초대받은 사람이 이메일이나 알림을 통해 **Accept invitation**을 눌러야 완료됩니다.

## 9. 오픈소스와 라이선스 (Open Source)

### 📖 역사와 개념
*   **배경**: 과거에는 프로그램 소스 코드를 공유하는 것이 당연했으나, 점차 기업들이 코드를 비공개(폐쇄적)로 관리하기 시작했습니다.
*   **자유 소프트웨어 운동**: 1983년 MIT의 리차드 스톨만(Richard Stallman)이 시작했습니다. "소프트웨어는 누구나 자유롭게 실행, 복사, 수정, 배포할 수 있어야 한다"는 정신입니다. (대표작: GNU 리눅스)
*   **오픈소스**: 소스 코드가 대중에 공개된 소프트웨어를 말합니다.

### 🌟 대표적인 오픈소스 프로젝트
*   **NumPy**: 파이썬 수치 계산 라이브러리
*   **Linux**: 리눅스 운영체제 커널
*   **MySQL**: 데이터베이스 관리 시스템
*   **React Native**: 페이스북의 모바일 UI 프레임워크
*   **TensorFlow**: 구글의 머신러닝 프레임워크

### ✅ 장점과 단점
| 장점 | 단점 |
| :--- | :--- |
| 무료로 사용할 수 있습니다. | 참여자가 적으면 신뢰성을 보장하기 어렵습니다. |
| 많은 개발자가 참여하여 코드의 신뢰도가 높습니다. | 문제가 생겼을 때 책임을 질 주체가 없습니다. |
| 질문과 답변을 주고받으며 성장할 수 있습니다. | |

> **주의사항**: 오픈소스 라이선스(License) 확인이 필수입니다. "공개된 코드 = 맘대로 써도 됨"이 아닙니다. (예: 사용 시 소스 공개 의무 등)

## 10. 효율성 & 팁 (Tips)

### 📦 임시 저장 (Stash)
작업을 하던 중 급하게 다른 브랜치로 넘어가야 하는데 커밋하기엔 애매할 때, 변경 사항을 잠시 서랍에 넣어두는 매우 유용한 기능입니다.
```bash
git stash          # 하던 작업 임시로 숨겨서 저장하기
git stash list     # 임시 저장된 목록 확인
git stash pop      # 가장 최근에 저장한 작업 불러오면서 목록에서 지우기
```

### 🚀 별명 붙이기 (Aliasing)
긴 명령어를 매번 치기 힘들 때 별명(alias)을 붙여 사용합니다.
```bash
git config --global alias.history 'log --pretty=oneline'
# 이제 git history라고 치면 git log --pretty=oneline이 실행됨
```

### 🔖 태그 (Tag)
커밋 중에서도 출시 버전(v1.0)처럼 중요한 의미가 있는 커밋에 이름표를 붙입니다.
```bash
git tag Version_1 [커밋ID]
git show Version_1
```

### 🔍 Git Blame (범인 찾기)
특정 파일의 코드를 누가, 언제 수정했는지 줄 단위로 보여줍니다.
```bash
git blame [파일명]
# 결과: [커밋ID] [작성자] [시간] [코드내용]
```
